bb_policy — Design & Architecture

bb_policy lets a robot run a learned behaviour — a neural network that looks at what the robot senses and decides what it should do — on the Erlang VM, in the same process tree that controls the hardware, with the safety system in the loop.

In one breath

Train a policy elsewhere (Python), export it to a portable model file (ONNX), drop it onto a Beam Bots robot, and run it at a fixed control rate — as a one-shot task, a named robot command, or a standing controller. A crashed or slow policy can't take the robot down, because it lives inside the BEAM's supervision model.

1 · Context & the problem

Most robot motion is explicitly programmed: "move joint 3 to 1.2 radians, then close the gripper." That works for repeatable, structured tasks. It falls apart for things like dexterous manipulation, grasping novel objects, or compliant motion — situations with too many contact states and variations to enumerate by hand.

For those, the field has moved to learning from demonstration: a human (or a simulator) shows the robot the task many times, and a neural network learns the mapping from what the robot perceives to what it should do. That mapping is called a policy. bb_policy is the piece that runs such a policy on real Beam Bots hardware.

Where this sits bb_policy is a satellite package of Beam Bots core (bb) — an Elixir framework for resilient robotics. Core owns the robot model, supervision tree, safety system, and actuator/sensor plumbing. bb_policy depends only on core's public API and adds the policy-running machinery. It deliberately does not do training, datasets, teleoperation, or vision — those are separate packages.

Glossary — terms used on this page

If you're coming from web/backend rather than robotics or reinforcement-learning, start here.

Policy
A function π: observation → action. Given what the robot senses, it returns what to do. Here it's almost always a neural network.
Observation
The policy's input: a bundle of numbers describing the robot's current state — joint angles, joint speeds, maybe a camera frame or force readings.
Action
The policy's output: the commands to send to the motors — e.g. target joint positions or velocities.
Joint / actuator
A joint is a movable connection (an elbow); an actuator is the motor that drives it. "Set joint to 1.2 rad" is an actuator command.
Control loop / rate (Hz)
A robot is controlled by repeating "sense → decide → act" many times a second. 20 Hz means 20 times per second (every 50 ms). That repeat is the control loop.
Inference
Running the trained neural network once to turn an observation into an action. (As opposed to training, which is how the network learned, done elsewhere.)
ONNX
Open Neural Network Exchange — a portable file format for trained models. Train in PyTorch/JAX, export to .onnx, run anywhere that has an ONNX runtime.
ACT / action chunking
"Action Chunking with Transformers" — a popular imitation-learning method. Instead of predicting one action, it predicts a short chunk of future actions at once. See §4.
Normalisation
Neural nets expect inputs scaled to a standard range (e.g. mean 0, std 1). Normalisation converts raw sensor units ↔ the scaled values the net was trained on. See §4.
Arm / disarm (safety)
A robot is armed when it's allowed to move and disarmed when motors are inhibited. Disarming is the emergency-stop concept. bb_policy never moves a disarmed robot.
BEAM / OTP
The Erlang virtual machine (BEAM) and its framework (OTP). It runs many tiny isolated processes with supervisors that restart crashed ones — which is why a faulty policy can't crash the whole robot.
GenServer / Behaviour
OTP building blocks. A GenServer is a stateful server process. A Behaviour is an interface (a set of callbacks a module must implement) — like a Java interface or a Rust trait.
Reactor
A Beam Bots workflow tool for sequencing steps (with rollback). bb_policy makes a policy usable as one step in such a workflow.

2 · What was built

Nine modules, each with a single responsibility. The split is deliberate: the behaviour (the interface) is separate from the implementation (ONNX), and the shared control step is separate from the three different ways to drive it.

The control cycle (one tick)

Every runtime repeats the same four-step cycle, captured once in BB.Policy.Step. The crucial detail is the safety gate: the cycle only ever drives motors while the robot is armed.

Three ways to run the same policy

A single policy can be driven in three ways, depending on whether the behaviour is a one-shot task, a step in a plan, or an always-on reflex. All three share BB.Policy.Step — they differ only in lifecycle and how the safety system reaches them.

4 · Key sub-systems explained

Action chunking (and why it needs a queue)

A naïve policy predicts one action per tick. ACT-style policies instead predict a chunk — a short sequence of future actions — in a single inference. This produces smoother motion and means you don't have to run the (relatively expensive) neural net on every single tick. But it raises a question: how do you turn one chunk of N actions into per-tick commands? bb_policy implements both standard answers.

Receding-horizon queue default

Run inference once, push the predicted chunk into a queue, then play it out one action per tick. When the queue empties, infer again. Fewer inferences, lower compute — the cheap, robust default.

tick 1 → infer → [a0, a1, a2]; play a0
tick 2 → play a1
tick 3 → play a2
tick 4 → infer again → …

Temporal ensembling opt-in

Infer every tick. Multiple overlapping chunks now each predict the current timestep; blend them with exponential weights wᵢ = exp(-coeff·age). Smoother and more reactive, at the cost of inferring every tick. Enable with temporal_ensemble_coeff:.

step1 = ( a@1.row0 · 1
        + a@0.row1 · e^(-coeff) )
        / (sum of weights)
Why both The queue is correct and cheap for most cases and is the safe default. Ensembling is what the original ACT work uses for the smoothest results; offering it (off by default) means we don't force the compute cost on everyone but support the high-quality path when wanted.

Normalisation — and why the runtime owns it

A neural net trained on data where joint angles averaged 0.1 rad with a spread of 0.5 expects inputs scaled the same way. The numbers needed to do that scaling (means, standard deviations, min/max) are a property of the training dataset, not the model graph.

The subtle part Tools like LeRobot strip normalisation out of the exported ONNX file and ship the statistics separately (as JSON). So the runtime — not the model — must re-apply them. BB.Policy.Normalizer loads those stats and scales observations in and actions out. It also guards against divide-by-zero on constant features (zero variance), which would otherwise produce NaN and crash the robot mid-motion.

Three strategies, chosen per key: :z_score (mean 0 / std 1), :min_max (scale a known range to [0,1] or [-1,1]), and :identity (passthrough).

Safety — non-negotiable

A learned policy is a black box that could, in principle, output anything. Two layers protect the hardware:

  • The gate: every tick checks BB.Safety.armed?/1 before applying commands. A robot that is disarmed — or disarms mid-episode — is never commanded; the episode ends with reason :disarmed. This is treated as a deliberate intervention, not an error to retry.
  • The envelope: the commands a policy emits still pass through core's actuator system, which enforces the robot's joint and velocity limits. A bad action can't drive a motor outside its configured range.

When run as a Command or Controller, the safety system reaches the policy through core's own callbacks (a disarm stops a command with reason :disarmed, which a Reactor workflow surfaces as {:halt, :safety_disarmed}).

5 · Decisions & trade-offs

These are the choices that shaped the package. Some were explicit (asked and answered); others were implicit (the obvious-in-hindsight calls made while building). Each lists what was decided, why, and what it costs.

D1 Public entry point is BB.Policy.run/4; BB.Motion.run_policy/4 is a deferred core PR explicit
Decision
Ship the imperative API as BB.Policy.run/4 (delegating to the Runner). The proposal's BB.Motion.run_policy/4 would require modifying a module that lives in bb core.
Why
A satellite package cannot add a function to a core module without a pull request to core. Keeping the entry point inside bb_policy makes the package self-contained and shippable today.
Trade-off
The convenience call shown in the original proposal isn't available yet; it's tracked as a separate core PR. Mild API-discoverability cost, zero coupling cost.
D2 A standalone Runner first; the BB.Controller path added alongside explicit
Decision
Build the bounded-episode Runner (plain GenServer) first, then add the continuous BB.Controller variant — sharing one control step.
Why
Episodic "run a task to completion" and standing "always-on reflex" are genuinely different lifecycles; both are real use-cases in the ecosystem. Extracting BB.Policy.Step means the cycle logic isn't duplicated.
Trade-off
Three runtime modules instead of one. Justified: each is thin, and the alternative (one over-configurable god-module) would be harder to reason about and test.
D3 Inference via Ortex.run/2 directly — not batched serving explicit
Decision
Call the ONNX runtime directly each tick, rather than routing through Nx.Serving's batched execution.
Why
Batched serving exists to amortise overhead across concurrent requests. A single robot at a fixed rate issues one inference at a time and never fills a batch — the batching timeout (default 100 ms) would only add latency to a 50 ms budget.
Trade-off
If a future use-case fans in multiple cameras or policies, batched serving may become worth it. Noted as a future path; not paid for now.
D4 Normalisation is owned by the runtime, not the model explicit
Decision
BB.Policy.Normalizer applies dataset statistics at inference time; the ONNX file is treated as not end-to-end.
Why
Real exports (LeRobot) strip normalisation from the graph and ship stats separately. Assuming the model is end-to-end would silently feed it wrong-scale inputs and produce garbage motion.
Trade-off
The operator must supply the stats file alongside the model. This matches how the export tools actually work, so it's honest rather than convenient-but-wrong.
D5 Target ACT first; first inference on a dev box, not Nerves explicit
Decision
Support ACT-class (static-shape) models first; diffusion / vision-language-action models are out of v1 scope. First real inference targets an x86/Mac dev box, not the robot's ARM board.
Why
ACT exports to ONNX cleanly; diffusion/VLA models have iterative loops that don't. The ARM/Nerves binary story is the single biggest integration risk (see R1) and shouldn't block proving the core works.
Trade-off
On-robot deployment is a separate, later phase. The dev-box path de-risks everything else first.
D6 ortex is an optional, gated dependency implicit
Decision
The behaviour, runner, command, controller, and normaliser have no ML-runtime dependency. ortex is optional and only fetched when ORTEX=1 is set; the two Ortex calls go through apply/3.
Why
Ortex builds a Rust NIF and downloads an ONNX runtime binary — needing a Rust toolchain. Forcing that on everyone who depends on bb_policy (and on CI) would be hostile. Most of the package is useful without it.
Trade-off
Using apply/3 sidesteps compile-time checks for those two calls (with a clear runtime guard and documented credo suppressions). The payoff: the package compiles --warnings-as-errors and passes dialyzer with no Rust present.
D7 A new ActuatorCommand struct, because core has no command type implicit
Decision
Introduce BB.Policy.ActuatorCommand (path / kind / value) as the return type of action_to_commands/3, with a dispatcher to core's actuator functions.
Why
The original proposal referenced a BB.Actuator.command() type that doesn't exist in core. Rather than invent coupling, a small declarative struct keeps policies decoupled from the actuator transport and lets the runtime own dispatch and the safety gate.
Trade-off
One more type to learn. But it's the seam that keeps the policy from knowing how commands are delivered. Named ActuatorCommand (not Command) to leave BB.Policy.Command free for the reactor wrapper.
D8 Policies signal completion via {:done, state} from act/2 implicit
Decision
A policy ends its own episode by returning the atom :done in place of an action.
Why
The behaviour needed a way for a policy to say "I've finished" without a separate callback. Overloading the existing act/2 return keeps the interface to five callbacks and threads cleanly through the shared Step.
Trade-off
It's a small overloading of one return value. The alternative (an extra done?/1 callback) would add interface surface for a binary signal.
D9 Generic command handler configured by options, not a code-gen macro explicit
Decision
BB.Policy.Command is one reusable use BB.Command handler configured via policy: / policy_opts: / rate_hz: in the DSL — rather than a macro that generates a dedicated command module per policy.
Why
One module, reused for any policy, with no metaprogramming. Easier to read, test, and reason about; it satisfies the same contract a Reactor step already invokes, so reactor-usability comes for free.
Trade-off
Slightly more verbose at the call site than a bespoke module name. Worth it to avoid generated code.
D10 Nix flake for a reproducible 1.19/OTP 28 toolchain implicit
Decision
Use a Nix dev shell (Elixir 1.19 / Erlang 28, plus Rust for the ONNX path) to build and test against the real toolchain the ecosystem targets.
Why
Core requires Elixir 1.19, newer than the host machine's default. Lowering the requirement would diverge from every other beam-bots package; Nix gives the exact toolchain without changing the project.
Trade-off
A Nix dependency for local dev (already present in the repo). The reward is that everything was verified against real bb, not stubs.

6 · Risk register

The honest list of what could bite, surfaced by an up-front review of the ML stack on the Erlang VM.

#RiskSeverityHow it's handled
R1On-robot (ARM/Nerves) deployment. Ortex ships no prebuilt binaries and pins a release-candidate ONNX runtime; you must cross-compile the Rust NIF and supply an ARM libonnxruntime.highDeferred to Phase 6 (the one not-yet-shipped phase). Dev-box path proven first so only the binary story remains.

The action-queue / ensembling logic isn't traceable; export is inference-only subgraphs, static shapes, with normalisation stripped out.highOwned by a Python-side export pipeline (outside this repo). The runtime expects a static-shape model + separate stats — which is what the tools actually produce (see D4).
R3Inference blocks the BEAM scheduler. A multi-millisecond Rust NIF on a normal scheduler hurts soft-real-time timing.mediumNoted for the on-target phase: isolate inference in its own process, cap the ONNX runtime's thread pool, and measure p99 (worst-case), not the mean.

R5"Any policy just loads" expectation. Diffusion/VLA models won't export cleanly today.lowScope is explicit (ACT first); a clear error rather than silent misbehaviour.

8 · Status & what's next

Shipped & verified: the behaviour, the shared control step, all three runtimes (Runner / Command / Controller), the ONNX implementation with both action-chunking regimes, normalisation, actuator dispatch, and telemetry — 61 tests + 2 doctests, all quality gates green against real bb 0.20.

Deliberately deferred (each noted in PROJECT_PLAN.md):

  • Phase 6 — on-robot deployment: cross-compiling the ONNX NIF for ARM and sourcing an ARM ONNX runtime; on-target latency/jitter measurement (R1, R3). This is a hardware/infra task for when a target device is available.
  • The BB.Motion.run_policy/4 delegate — belongs in a PR to the bb core repo (D1).
  • End-to-end tests against a live, supervised robot — the runtime callbacks are tested directly and against real core APIs; a full DSL-robot + reactor harness is the remaining integration layer.
  • Multi-input models (e.g. camera + joint state) and a sensor-collection pipeline for vision.
